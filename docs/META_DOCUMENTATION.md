# Documentation System Guide

> **📍 Purpose:** AI agent directives for maintaining documentation consistency.
> Prevents drift through single sources of truth and temporal separation.

---

## Core Principles

### 1. Single Source of Truth (SSOT)

Every piece of information has EXACTLY one canonical location. Never duplicate.

### 2. Temporal Separation

Files organized by update frequency:

| Frequency  | Files                                 | Timestamps?            |
| ---------- | ------------------------------------- | ---------------------- |
| Never      | ARCHITECTURE.md, DATA_SCHEMA.md       | ❌ No                  |
| Quarterly  | TESTING_STRATEGY.md, CLAUDE.md        | ❌ No                  |
| Per Phase  | CURRENT_PHASE.md, DEPLOYMENT_GUIDE.md | ✅ Date only           |
| Per Commit | METRICS.md                            | ✅ Date+time (minutes) |

**Note:** Task tracking migrated to Linear (2025-11-03).

### 3. Fact vs Opinion

- **Facts** (verifiable) → Generated by scripts
- **Observations** (judgment) → Manual documentation

### 4. Conflict Resolution

When information conflicts:

```
1. Generated data > Manual claims
2. Recent date > Old date
3. Specific location > General
4. Lower in tree > Higher in tree
```

## When to Update What

### Changed Rust types?

```bash
just types-sync
# Schema + TS types auto-generated
# Manual update needed: docs/DATA_SCHEMA.md if field semantics changed
```

### Completed a phase?

**Update:** `docs/CURRENT_PHASE.md`
**Add:** `last_updated: 2025-10-30` (date only)

### Added/removed tests?

**Update:** Nothing (auto-generated on next commit)

### Started new task?

**Update:** Linear project (create or update issue)

### Changed architecture?

**Update:** `docs/ARCHITECTURE.md` (rare)

### Changed testing approach?

**Update:** `docs/TESTING_STRATEGY.md` (quarterly)

## Anti-Patterns (NEVER Do This)

❌ **Duplicating information "for convenience"**

```markdown
# Bad: In CLAUDE.md

Tests: 361 (see METRICS.md)

# Good: In CLAUDE.md

For test counts, see docs/METRICS.md
```

❌ **Timestamps without date format**

```yaml
# Bad
last_updated: 2025-10-28T14:30:52.123Z

# Good (date only for most docs)
last_updated: 2025-10-28

# Good (date+time for metrics only)
Generated: 2025-10-28T14:30
```

❌ **Manually editing generated files**

- Never edit `docs/METRICS.md` by hand
- Fix generation script instead

❌ **Claiming numbers without verification**

```markdown
# Bad

We have 236 Rust tests.

# Good

For current test counts, see docs/METRICS.md (auto-generated).
```

## Good Patterns (ALWAYS Do This)

✅ **Link to canonical source**

```markdown
For [information], see [canonical-file.md]
```

✅ **Generate dynamic data from commands**

```bash
# Parse actual output
RUST_COUNT=$(grep "passed" .logs/rust-tests.log | awk '{sum+=$1} END {print sum}')
```

✅ **Add timestamps to manual updates**

```yaml
---
last_updated: 2025-10-28
---
```

✅ **Verify docs in pre-commit**

- `scripts/verify-docs.sh` catches drift

## Automation

### `scripts/update-metrics-from-logs.sh`

**Trigger:** Every commit (pre-commit hook)
**Generates:** `docs/METRICS.md`
**Input:** Test logs from `.logs/rust-tests.log` and `.logs/ts-tests.log`

### `scripts/verify-docs.sh`

**Trigger:** Every commit (pre-commit hook)
**Checks:**

1. No test counts in CLAUDE.md
2. METRICS.md <24h old
3. Timestamps are date-only (or date+time for metrics)
4. No duplicate information

## Quick Reference for AI Agents

### Before Work

1. Ask: "Which Linear issue?"
2. Fetch issue details from Linear
3. Check `.claude/CLAUDE.md` → Find docs
4. Read `docs/CURRENT_PHASE.md` → Current phase
5. Read component CLAUDE.md if in subdirectory

### During Work

- Use `just test` (fast feedback)
- Use `just check` (type validation)
- Use `just types-sync` (after Rust changes)

### After Work

1. Update Linear issue status
2. Update `docs/CURRENT_PHASE.md` (if major milestone)
3. Run `just test` (auto-regenerates METRICS.md)
4. Commit (hooks verify everything)

### Critical Rules

- ⚠️ NEVER edit METRICS.md manually
- ⚠️ NEVER duplicate information
- ⚠️ ALWAYS link to canonical source
- ⚠️ ALWAYS use date-only timestamps (except metrics)
- ⚠️ ALWAYS verify before claiming numbers

## Generic Template

This system works for ANY project:

```
.claude/CLAUDE.md              # Router
docs/
  ├── ARCHITECTURE.md          # Immutable design
  ├── CURRENT_PHASE.md         # Active phase
  ├── TESTING_STRATEGY.md      # Philosophy
  ├── METRICS.md               # Generated metrics
  ├── LINEAR_WORKFLOW.md       # Linear task tracking guide
  ├── ANALYTICS_ARCHITECTURE.md # Event tracking
  ├── BUILD_PIPELINE.md        # Build system
  ├── CODEBASE_REVIEW.md       # Tech debt audit
  ├── DEPLOYMENT_GUIDE.md      # Deployment how-to
  ├── POSTHOG_DASHBOARD_SPEC.md # Dashboard specs
  └── META_DOCUMENTATION.md    # This guide
scripts/
  ├── update-metrics-from-logs.sh
  └── verify-docs.sh
```

**Core principles apply everywhere:**

1. Single source of truth
2. Temporal separation
3. Generated facts
4. Manual observations
5. Automated verification
